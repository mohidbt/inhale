## 0. Non-Negotiable Testing Rules

These rules exist because the last pass shipped seven broken features while all unit/E2E tests "passed." Violating them = the same failure.

### 0.1 Ban list — never do these

1. **Never mock the API route you are claiming to test.**
   - If Task X implements `/api/foo`, then `page.route("**/api/foo", ...)` is **forbidden** in the E2E test for Task X.
   - Mocking is only allowed for APIs *adjacent* to the feature (e.g., mock `/api/third-party/embed` when testing UI, not the endpoint you wrote).
2. **Never assert intermediate state as proof of feature.**
   - "Input has value" ≠ "feature works."
   - "Component mounted" ≠ "highlight visible."
   - "Request sent" ≠ "search ran."
3. **Never conclude a feature works from unit tests only.** Unit tests prove a function's shape. Features live in the DOM of a real page.
4. **Never declare a phase DONE without a manual DevTools walk-through** that demonstrates every user-visible outcome listed in the Definition of Done.

### 0.2 Must-do list for every feature test

- **Use a real PDF fixture** (`e2e/fixtures/test_real_paper.pdf`) — not `test.pdf` — unless the feature is page-count agnostic.
- **Assert the user-visible outcome in DOM**, as a user would observe it. The outcome must be specified *before* writing the test (see "User-visible outcome" field on every issue below).
- **Real backend, not mocks.** Use `INHALE_STUB_EMBEDDINGS=1` (already set by playwright config) to make embeddings deterministic; hit the real chat/outline/search paths.
- **Screenshot on failure AND success.** `await page.screenshot({ path: ... })` after the critical assertion. Commit failing screenshots into `e2e/__failures__/` for review.
- **DevTools MCP walkthrough** (Chrome DevTools MCP tools) for every issue: upload → perform action → verify in live DOM → capture screenshot → attach to PR.

### 0.3 Test-review gate

Before merging any fix below, the reviewer subagent MUST check:

```
- [ ] The test does NOT mock the endpoint the task modifies.
- [ ] The test uses a real PDF.
- [ ] The test asserts a visible DOM outcome matching the "User-visible outcome" field.
- [ ] A DevTools walkthrough screenshot is attached to the PR.
- [ ] Running `npm run build && npm run test && npx playwright test` all pass.
- [ ] The tester actually loaded the reader page in a browser once, manually.
```

Fail any box → not mergeable.

### Issue 1 — Highlights never render on PDF

**Symptom:** User saves a highlight → nothing visible on page. Highlights sidebar lists it, but the canvas overlay is empty.

**Root cause:** `src/app/(reader)/reader/[documentId]/reader-client.tsx` passes `markers={markers}` to `<PdfViewer>` but never passes `userHighlights`. The data is fetched only inside `HighlightsSidebar` and never lifted.

**Fix:**
1. In `reader-client.tsx`, fetch highlights at the page level (or lift the existing fetch out of the sidebar).
2. Shape the data to `UserHighlight[]` (see `user-highlight-layer.tsx` type).
3. Pass as `userHighlights={...}` to `<PdfViewer>`.
4. Re-fetch when `refreshKey` bumps (after save/delete).

**Files:**
- Modify: `src/app/(reader)/reader/[documentId]/reader-client.tsx`
- Modify: `src/components/reader/highlights-sidebar.tsx` (accept highlights + loading as props, lift fetch)
- New hook optional: `src/hooks/use-user-highlights.ts`

**User-visible outcome:**
After selecting text, choosing yellow in the toolbar, releasing the click:
- A yellow translucent rectangle appears **on the PDF page** behind the selected words.
- Reloading the page: the rectangle is still there.

**Test (must follow §0):**
```ts
// e2e/highlights-render.spec.ts — NEW
// Real PDF. Real backend. Assert DOM.
test("saving a highlight renders a colored overlay on the PDF", async ({ page }) => {
  await signUpAndLogin(page);
  const { id } = await uploadTestPdf(page, "test_real_paper.pdf");
  await page.goto(`/reader/${id}`);
  await expect(page.locator("canvas").first()).toBeVisible({ timeout: 10_000 });

  // Select a word on page 1 via DOM range.

    const span = document.querySelector(".react-pdf__Page__textContent span");
    if (!span) throw new Error("no text span");
    const r = document.createRange();
    r.selectNodeContents(span);
    const sel = getSelection()!;
    sel.removeAllRanges(); sel.addRange(r);
    document.dispatchEvent(new Event("selectionchange"));
  });
  await page.getByTitle("yellow").click();

  await expect(page.locator("[data-highlight-id]")).toBeVisible();
  await page.reload();
  await expect(page.locator("canvas").first()).toBeVisible({ timeout: 10_000 });
  await expect(page.locator("[data-highlight-id]")).toBeVisible();
});
```

### Issue 2 — Outline has no Contents tab for most PDFs; "Pages" is text-only

**Symptom:** Scientific PDFs show only a text list "Page 1, Page 2…". No chapter/section tree. User expects a Chrome-like experience: thumbnails in Pages tab, section tree in Contents tab.

**Root cause:**
- `OutlineSidebar` Contents tab only appears if `pdf.getOutline()` returns something. Most research PDFs have no embedded bookmarks → no Contents.
- Pages tab renders plain text buttons, not thumbnails.
- The previous `/api/documents/[id]/outline` section-extraction endpoint was deleted in Task 31 without a replacement.

**Fix:**
1. **Pages tab → thumbnails.** Render a small PDF.js canvas thumbnail per page (lazy, via IntersectionObserver — never render all upfront on 60-page papers). Use `pdfDoc.getPage(n).render({ canvasContext, viewport })` at a viewport scale producing ~120px width.
2. **Contents tab — fallback to computed outline** when `pdf.getOutline()` is null/empty.
   - Restore a lean version of section extraction (heuristic: look at font-size / bold spans on the text layer, pick top 2 size tiers as H1/H2). Keep it client-side to avoid re-introducing the server endpoint. Reference: PDF.js `getTextContent` + styling info per page.
   - If heuristic produces nothing, show a helpful empty state ("This PDF has no table of contents") — never a blank tab.
3. Always render both tabs; disable or show empty state in Contents when nothing exists. Never hide the tab.

**Files:**
- Modify: `src/components/reader/outline-sidebar.tsx`
- New: `src/components/reader/page-thumbnail.tsx`
- New: `src/hooks/use-computed-outline.ts` (client-side section extraction)

**User-visible outcome:**
- Pages tab: visible miniature PDF thumbnails, clickable to scroll.
- Contents tab: either a section tree (native or computed) or a clear empty state — tab is always present.

**Test:**

### Issue 3 — Cmd+F search does nothing

**Symptom:** FindBar UI opens, typing a query finds nothing visually.

**Root cause:**
- `usePdfFind` calls `controller.executeCommand("find", {...})` — internal API. Current PDF.js expects `eventBus.dispatch("find", { type: "", query, ... })`.
- Even if the controller computes matches, react-pdf's TextLayer is **not** registered with `PDFFindController`, so no DOM highlight.

**Fix (pick one — simpler wins):**

**Option A (recommended — DOM search, no PDF.js wiring):**
1. Rewrite `usePdfFind` to search the already-rendered `.react-pdf__Page__textContent span` elements. Walk spans, mark matches with `<mark class="find-match">` wrappers or by toggling a data attribute + CSS highlight.
2. Track current match index; `next()` / `prev()` scroll the matched element into view and toggle `.find-match--current`.
3. Pros: no PDF.js integration required, works with virtualized pages (re-run on newly rendered pages).
4. Cons: only finds text on currently-rendered pages — handle virtualization by forcing render of target page on match hit.

**Option B (PDF.js wiring):**
1. Bridge react-pdf's TextLayer with `PDFFindController` via a custom TextLayerBuilder. Significant react-pdf source work. Avoid unless A is blocked.

Go with A. Keep `FindBar` UI unchanged (match-case, next/prev, close).

**Files:**
- Rewrite: `src/hooks/use-pdf-find.ts`
- Add CSS: `src/app/globals.css` — `.find-match { background: rgba(253,224,71,0.6) } .find-match--current { background: rgba(234,179,8,0.9) }`

**User-visible outcome:**
Type "rare" → every visible occurrence of "rare" gets a yellow background; Enter/next cycles; current match goes darker yellow and the page scrolls to it.

**Test:**

### Issue 4 — Comment popup collapses instantly; Comments have no tab

**Symptom:** Click "Comment" in selection toolbar → textarea flashes and disappears.
**Also:** users want a dedicated Comments tab; currently comments only show inline inside the Highlights sidebar.

**Root cause:**
`SelectionToolbar` is conditionally rendered based on `selection` state. Clicking the Comment button fires `selectionchange` (selection clears because focus shifts to the button) → `selection` becomes null → `<SelectionToolbar>` unmounts.

**Fix:**
1. **Decouple toolbar lifetime from window selection.** When user clicks a toolbar action, snapshot `selection` into local state inside `ReaderClient` (`const [activeSelection, setActiveSelection] = useState(null)`). While `activeSelection` is set, keep rendering the toolbar even if the window selection clears. Clear `activeSelection` on Save/Cancel/Escape.
2. **Comments tab.** Add a new `CommentsSidebar` component and a toolbar button. Source: reuse the highlights table, filter to ones with non-null `comment`. Show as a list sorted by page then createdAt. Clicking a comment scrolls to the highlight.

**Files:**
- Modify: `src/components/reader/selection-toolbar.tsx` (no longer reads from window selection; takes stable props)
- Modify: `src/app/(reader)/reader/[documentId]/reader-client.tsx` (snapshot selection; manage `activeSelection`)
- New: `src/components/reader/comments-sidebar.tsx`
- Modify: `src/components/reader/reader-toolbar.tsx` (add Comments button)

**User-visible outcome:**
- Select text → toolbar appears → click Comment → textarea stays open until Save or Cancel.
- Save → toolbar closes → comment visible in Comments tab AND on the highlight card.
- Comments tab lists all commented highlights.

### Issue 5 — AI chat context is architecturally wrong

**Symptom:** Ask AI on a highlighted sentence dumps the sentence alone into the input. Even normal chat ("what does this paper say?") returns "I only have access to page 1."

**Root causes:**
1. `handleAskAi` seeds chat input with `selection.text` only. No page wrapping, no dedicated system prompt.
2. Chat route's system prompt: "Use ONLY the provided context. The user is currently viewing page X." Combined with top-6 embedding chunks being near page X, the model concludes it only has page X.
3. No full-page-text injection. Retrieval is purely embedding-based.
4. No "scope mode" distinction between page-scoped ("explain this sentence") and paper-scoped ("summarize this paper") questions.

**Fix:**

**A. Chat API — full-page + broader retrieval + better prompt.**
Files: `src/app/api/documents/[id]/chat/route.ts`

1. Accept a new `scope` field: `"page" | "selection" | "paper"` (default `"paper"`).
2. Accept `selectionText?: string` and `pageNumber?: number`.
3. For `scope === "selection"` or `"page"`: fetch the full text of `pageNumber` from `document_chunks` (or from a `document_pages` table if present — verify schema) and inject as "Current page:". Still run RAG for supporting context (top 4 across the whole doc).
4. For `scope === "paper"`: run broader retrieval — top-8 chunks *spread across pages* (enforce `DISTINCT ON (page_start)` or a diversity penalty), plus the paper's abstract/first-page content as anchor.
5. Rewrite system prompt to separate "Primary focus" (selection/page) from "Supporting context" (chunks) and stop saying "ONLY the provided context" — say "prefer the provided material; if it is insufficient, say what is missing rather than refusing."
6. Log retrieval output for every request (redacted in prod) so we can diagnose future context issues.

**B. ChatPanel + ReaderClient — scope-aware seed.**
Files: `src/components/reader/chat-panel.tsx`, `src/app/(reader)/reader/[documentId]/reader-client.tsx`, `src/hooks/use-chat.ts`

1. Replace `seed: { text, nonce }` with `seed: { text, pageNumber, scope, nonce }`.
2. "Ask AI" action sends `scope: "selection"` + `selectionText` + `pageNumber`, not just raw text.
3. `sendMessage` forwards these to the API.
4. Chat input can still be edited — the scope propagates with the message.

**User-visible outcome:**
- Highlight sentence → Ask AI → send "" (empty — or pre-filled "Explain this") → answer references the sentence and the surrounding page, not a refusal.
- "What does this paper say?" → answer covers multiple pages, cites e.g., (p. 2), (p. 5), (p. 7).
- "Summarize page 3" with viewport on page 3 → answer reflects page 3 content specifically.

**Tests (DO NOT MOCK `/api/documents/*/chat`):**

### Issue 6 — Docking doesn't dock; narrow sidebars re-align badly

**Symptom:** Clicking the `⋮` dock menu on a sidebar shows right/bottom/left but picking "bottom" or "left" does nothing useful. Resizing narrow causes header items (AI Assistant / New / History) to wrap and the dock `⋮` to float over the title.

**Root cause:**
1. `DockableSidebar` was hand-rolled. All sidebars are siblings in a single `flex-row` parent. Setting `height: X, width: 100%` on a flex-row child does not dock it to the bottom — it just takes full horizontal space.
2. The dock `⋮` is absolutely positioned at `right-1 top-1` on every sidebar, overlapping the panel's own header at narrow widths.
3. Plan Task 27 specified `react-resizable-panels` — wasn't used.

**Fix:**
1. **Use `react-resizable-panels` as originally specified.** Replace the hand-roll in `dockable-sidebar.tsx` with `<PanelGroup direction="horizontal">` for side sidebars and `<PanelGroup direction="vertical">` for bottom-docked panels.
2. **Change layout root.** Wrap `reader-client.tsx` main area in a `PanelGroup direction="vertical"` so "bottom" dock moves a sidebar below the PDF row instead of beside it.
3. **Move dock menu into the sidebar's own header.** Remove the absolute-positioned `⋮`. Each sidebar (HighlightsSidebar, ChatPanel, CitationsSidebar, OutlineSidebar, CommentsSidebar) gains a `slot` for a docking control that the panel wrapper passes in. Renders as a small icon inside the header row — no overlap.
4. Minimum widths: enforce 280px min so the header never wraps.

**Files:**
- Install (if not present): `react-resizable-panels`
- Rewrite: `src/components/reader/dockable-sidebar.tsx`
- Modify: `src/app/(reader)/reader/[documentId]/reader-client.tsx`
- Modify: each sidebar's header (pass dock control through children or prop)

**User-visible outcome:**
- Pick "bottom" → sidebar moves below the PDF viewer, spanning full width.
- Pick "left" → sidebar moves to the left of the PDF viewer, reorders DOM, scrollbar on correct side.
- Resize to narrow — header stays on one line down to 280px, then blocks resize.
- Dock control lives inside the sidebar header, no overlap.

**Test:**
